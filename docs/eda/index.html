Architecture

真実は :eda.* datom と artifact CID。UI と LLM はその上の操作面であり、sign-off authority ではない。

kotoba-lang source

.kotoba / .cljc で EDA generator、constraint transform、workflow FSM、Datalog view を記述する。

kami engine viewer

schematic graph、GDS/OASIS tile、waveform、STA critical path、DRC/LVS overlay、package/thermal view を WebGPU で表示する。

murakumo jobs

synthesis、simulation、DRC、LVS、STA、SPICE、large report analysis、LLM inference を scale-to-zero job として実行する。

kotoba store

spec、netlist、layout、report、waiver、review、release bundle を datom 化し、大容量 artifact は CID で参照する。

EDA Flow

最初の対応は digital mixed-signal 寄り。既存 EDA tool を置き換えず、adapter 境界で datom/CID に正規化する。

Spec — requirements、interface、clock/reset/power domain、test intent を EDN datom 化。
Design — .kotoba / .cljc generator、RTL、SPICE、constraint、testbench。
Run — synthesis / P&R / simulation / DRC / LVS / STA / SPICE を murakumo job で実行。
Review — report diff、root-cause、fix candidate、waiver proposal を :eda.review/* に保存。
Release — required checks、tool versions、input/output CID、waiver、approval chain を bundle 化。

LLM Integration

LLM は copilot。実 EDA report が authoritative で、LLM 出力は必ず proposal datom になる。

spec-agent

natural language spec と既存 interface から EDN spec proposal を作る。human approve 必須。

rtl-agent

spec、interfaces、constraints から patch / generator proposal を作る。tests required。

verify-agent

reports、waveform summary、run history から root-cause と next-run plan を出す。既定 read-only。

release-agent

reports と waivers から sign-off checklist diff を作る。自分では承認できない。

Safety Boundary

Financial — large CPU/GPU run は quota + approval。
Confidential — PDK / customer data の外部送信は policy fail-closed。
External disclosure — foundry / vendor upload は explicit approval。
Quality — release candidate は required reports green が条件。
LLM patch — code-change は tests + human approve。
